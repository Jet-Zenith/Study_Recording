# Git 核心实战指南：从代码管理到高效协作

## 1. 为什么我们需要 Git？

在没有 Git 之前，我们管理代码或文档的痛点通常是这样的：

- `Train_model_v1.py`
- `Train_model_v2_Attention.py`
- `Train_model_v3_Final.py`
- `Train_model_v4_Final_True_师兄修改版.py`

当你想要找回三天前那个“还能正常运行”的代码时，往往已经无从下手。Git 就是为了解决这个问题而生的**分布式版本控制系统**。它不仅能帮你清晰地记录每一次代码的变更历史，还能让你在不同版本之间自由穿梭，更重要的是，它是你与课题组其他成员协同开发的基石。

------

## 2. 理解 Git 的“四个工作区”

Git 的运作逻辑建立在四个核心区域之上。理解了这四个区域的数据流向，Git 的命令就不再是死记硬背的字母了。

1. **工作区 (Working Directory)：** 你当前正在编写代码的目录。比如你的 Spring Boot 项目文件夹，或者存放 `train_attention.py` 的目录。
2. **暂存区 (Staging Area)：** 类似于一个“购物车”。当你写完一段代码准备提交时，先把它放进暂存区。
3. **本地仓库 (Local Repository)：** 点击“结算”后的历史记录存放地。代码一旦提交到这里，就生成了一个不可篡改的版本快照。
4. **远程仓库 (Remote Repository)：** 托管在云端（如 GitHub, Gitee, GitLab）的仓库，用于备份和团队协作。

------

## 3. 基础操作流水线：日常开发的四步曲

假设你今天开始了一个新的独立项目，以下是你每天都会用到的基础命令：

### 步骤一：初始化与查看状态

- **`git init`**：在当前文件夹打开终端输入此命令，将普通文件夹变成受 Git 管理的仓库。
- **`git status`**：这是你最常用的命令，随时查看当前仓库的状态（哪些文件被修改了，哪些在暂存区）。

### 步骤二：加入暂存区（挑选商品）

如果你新增了一个用于处理业务异常的类，现在需要将其纳入版本控制：

- **`git add DataConflictException.java`**：只将特定的文件放入暂存区。
- **`git add .`**：将当前目录下所有修改过或新增的文件一次性放入暂存区。

### 步骤三：提交到本地仓库（结账存档）

- **`git commit -m "feat: 新增 DataConflictException 异常处理"`**
  - `-m` 后面跟着的是本次提交的说明。良好的提交信息（Commit Message）至关重要。建议使用规范的前缀，例如 `feat:` (新功能)、`fix:` (修复 Bug)、`docs:` (文档更新)。

### 步骤四：推送到远程仓库（云端备份）

- **`git push origin main`**：将本地 `main` 分支的代码推送到远程仓库的 `main` 分支。

------

## 4. 分支管理 (Branching)：开启平行宇宙

分支是 Git 最强大的特性。它允许你在不影响主线代码的情况下，安全地开发新功能或进行破坏性实验。

**场景体验：** 你的主分支 (`main`) 运行着一个非常稳定的多智能体协作系统。现在你想尝试加入一个新的 `ProblemExtractAgent`，但不确定能否成功。

1. **创建并切换分支：**

   `git checkout -b feat-problem-extract-agent`

   （这相当于克隆了一个当前代码的平行宇宙，你在这里怎么折腾都不会影响 `main` 分支。）

2. **在分支上开发：** 编写你的异步逻辑，然后正常执行 `add` 和 `commit`。

3. **切回主分支：**

   `git checkout main`

   （瞬间，你的代码目录又变回了原来那个稳定的版本。）

4. **合并分支：**

   `git merge feat-problem-extract-agent`

   （如果你在分支上的测试完美通过，使用此命令将新代理的代码合并回主线。）

------

## 5. 团队协作与防坑指南

当你的课题组共同维护一个项目时，远程协作就变得频繁。

- **`git clone <远程仓库地址>`**：新加入项目时，将师兄建好的远程仓库完整下载到本地。
- **`git pull origin main`**：**关键习惯！** 每天开始写代码前，或者在准备 `push` 自己的代码前，先执行 `pull`，拉取远程仓库最新的代码到本地，这能极大减少代码冲突（Merge Conflict）的概率。

------

## 6. `.gitignore` 的艺术：不要把垃圾塞进仓库

Git 应该只追踪代码和配置文件。编译生成物、环境依赖包以及庞大的数据集**绝对不能**提交到 Git 中，否则会让仓库变得极其臃肿。

在项目的根目录下创建一个名为 `.gitignore` 的文件，并在里面配置需要忽略的规则。例如：

```Plaintext
# Java / Spring Boot 项目需要忽略的文件
target/
*.class
.idea/
*.iml

# Python / 深度学习项目需要忽略的文件
__pycache__/
.conda/
venv/
*.pth
*.pt
# 忽略海量的测试数据集
dataset/12k_drive_end_bearing/
dataset/titanic/*.csv
```
