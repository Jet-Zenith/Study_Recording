# Java反射机制 学习笔记

## 一、 深度理解：反射的底层物理基石（JVM 视角）

要真正理解反射，必须先明白 Java 类的生命周期和 JVM 的内存结构。

### 1. 类的加载与 `Class` 对象

当我们写好一个 `User.java` 文件并编译后，会生成一个 `User.class` 字节码文件。

1. **加载阶段**：JVM 的类加载器（ClassLoader）读取 `User.class` 字节码文件，将其中的类信息（类名、修饰符、方法、字段等）加载到内存中的**方法区（Method Area）**。
2. **生成 Class 对象**：同时，JVM 会在堆内存（Heap）中自动创建一个 `java.lang.Class` 类型的对象。这个对象就像是一面镜子，它完美地映射了方法区里该类的所有结构。

> 💡 **核心结论**：
>
> 每个类在 JVM 中**有且仅有**一个对应的 `Class` 对象。反射的所有操作，本质上就是通过这个堆里的 `Class` 对象去访问方法区里的数据。

### 2. 获取 Class 对象的三种方式——底层差异

虽然有三种方式获取 `Class` 对象，但它们触发的类加载时机是不同的：

```Java
// 方式一：Class.forName("com.xyz.User")
// 会触发类的【初始化】。如果该类有 static 静态代码块，会被立即执行！
Class<?> clz1 = Class.forName("com.xyz.User");

// 方式二：User.class
// 属于类字面量。仅仅让 JVM 把类【加载】进来，绝对【不会】触发类的静态代码块初始化。
Class<User> clz2 = User.class;

// 方式三：user.getClass()
// 对象都已经 new 出来了，说明类早就在堆里，并且已经完成了初始化。
User user = new User();
Class<? extends User> clz3 = user.getClass();

System.out.println(clz1 == clz2); // true
System.out.println(clz2 == clz3); // true 
```

## 二、 核心 API 详解与代码实战

假设我们有一个用于测试的复杂类 `Person`：

```Java
package com.xyz;

public class Person {
    public String name;
    private int age;
    
    // 无参构造
    public Person() {}
    
    // 私有有参构造
    private Person(String name, int age) {
        this.name = name;
        this.age = age;
    }
    
    // 公有方法
    public void Walk(int distance) {
        System.out.println(name + " 走了 " + distance + " 米");
    }
    
    // 私有带返回值方法
    private String getSecret(String password) {
        if ("123".equals(password)) return "其实我有两套房";
        return "无权访问";
    }
}
```

### 1. 构造方法（Constructor）的深度操作

`Constructor` 对象代表类的构造方法。

| **API 方法**                                         | **描述**                                   |
| ---------------------------------------------------- | ------------------------------------------ |
| `getConstructors()`                                  | 仅获取所有 **public** 的构造方法           |
| `getDeclaredConstructors()`                          | 获取**所有**构造方法                       |
| `getConstructor(Class<?>... parameterTypes)`         | 根据参数类型获取特定的 **public** 构造方法 |
| `getDeclaredConstructor(Class<?>... parameterTypes)` | 根据参数类型获取特定的**任意**构造方法     |

```Java
Class<?> clz = Class.forName("com.xyz.Person");

// 1. 调用私有构造方法
Constructor<?> declaredConstructor = clz.getDeclaredConstructor(String.class, int.class);

// 2. 必须解除私有特权
declaredConstructor.setAccessible(true); 

// 3. 实例化
Object personObj = declaredConstructor.newInstance("张三", 25);
```

### 2. 属性（Field）的深度操作

| **API 方法**                                              | **描述**                                 |
| --------------------------------------------------------- | ---------------------------------------- |
| `getFields()` / `getDeclaredFields()`                     | 获取多个属性（前者公有，后者所有）       |
| `getField(String name)` / `getDeclaredField(String name)` | 获取指定名称的属性（前者公有，后者所有） |

```Java
// 假设已经拿到上面创建的 personObj
// 修改公有属性
Field nameField = clz.getField("name");
nameField.set(personObj, "李四"); // 将 personObj 对象的 name 修改为 "李四"

// 修改私有属性
Field ageField = clz.getDeclaredField("age");
ageField.setAccessible(true); // 破除私有限制
ageField.set(personObj, 30);  // 修改为 30
```

### 3. 方法（Method）的深度操作

| **API 方法**                                         | **描述**                               |
| ---------------------------------------------------- | -------------------------------------- |
| `getMethod(String name, Class<?>... pTypes)`         | 获取指定的 public 方法（含父类继承的） |
| `getDeclaredMethod(String name, Class<?>... pTypes)` | 获取本类指定的任意方法（不含父类）     |

```Java
// 调用公有无返回值方法 walk(int)
Method walkMethod = clz.getMethod("Walk", int.class);
walkMethod.invoke(personObj, 500); // 输出：李四 走了 500 米

// 调用私有带返回值的方法 getSecret(String)
Method secretMethod = clz.getDeclaredMethod("getSecret", String.class);
secretMethod.setAccessible(true);
// 接收返回值
String result = (String) secretMethod.invoke(personObj, "123");
System.out.println(result); // 输出：其实我有两套房
```

## 三、 反射的高级应用场景

反射绝不仅仅是用来应付考试的，在日常架构设计中，以下两个场景它几乎是统治地位。

### 1. 破坏泛型擦除（黑客手段）

Java 的泛型是“伪泛型”，只在**编译期**有效，编译成字节码后泛型会被擦除（变成 `Object`）。利用反射，我们可以在运行时绕过泛型的限制。

```Java
List<Integer> list = new ArrayList<>();
list.add(111);
// list.add("Hello"); // 编译报错！

// 利用反射绕过检查
Class<? extends List> clz = list.getClass();
Method addMethod = clz.getMethod("add", Object.class);

// 成功把 String 类型塞进了 Integer 集合中！
addMethod.invoke(list, "我是通过反射混进来的字符串");

System.out.println(list); // 输出: [111, 我是通过反射混进来的字符串]
```

### 2. 模拟通用万能的“对象拷贝 / 框架核心”

写一个精简版的 `ObjectToString` 工具，无论传入什么对象，都能打印出它的属性名和属性值（这就是很多 JSON 序列化框架的雏形）。

```Java
public static void printObjectFields(Object obj) throws Exception {
    Class<?> clz = obj.getClass();
    System.out.println("--- 正在解析类：" + clz.getName() + " ---");
    
    Field[] fields = clz.getDeclaredFields();
    for (Field field : fields) {
        field.setAccessible(true);
        // 获取属性名
        String name = field.getName();
        // 获取该属性在当前传入对象中的具体数值
        Object value = field.get(obj);
        System.out.println(name + " 属性的值是: " + value);
    }
}
```

## 四、 进阶必知：反射的性能开销与优化

面试官经常会问：“反射很方便，但为什么说它慢？慢在哪里？”

### 1. 为什么慢？

- **JIT 无法优化**：现代 JVM 依靠 JIT（即时编译器）将热点代码编译为机器码，并进行指令重排、内联等深度优化。而反射代码是动态解析的，JVM 无法对其进行预判和内联优化。
- **安全检查开销**：每次调用 `Method.invoke` 或 `Field.get`，JVM 都要检查方法/属性的可见性、参数类型匹配、权限等。
- **变长参数的数组创建**：`invoke(Object obj, Object... args)` 的参数是可变参数，每次调用都要在堆里创建一个 `Object[]` 数组。

### 2. 怎么优化？

1. **缓存 Class、Method、Field 对象**：绝对不要在频繁调用的循环里写 `Class.forName()`。应该在类初始化时，用一个 `Map` 将这些反射对象缓存起来复用。
2. **合理关闭安全检查**：如果必须频繁使用反射，调用一次 `setAccessible(true)` 可以关闭安全检查，这能提升数倍的反射执行效率（即使是 public 元素，调用它也能提升速度）。
3. **高级替代方案**：在 JDK 7 之后，引入了 `java.lang.invoke.MethodHandle`（方法句柄），其性能比传统反射更接近直接调用。
